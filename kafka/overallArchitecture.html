PartitionsUnit of parallelism. More partitions = higher throughput but more file handles and replication overhead. Partition count can only increase, never decrease.
SegmentsEach partition is split into rolling segment files on disk (.log + .index). Older segments are eligible for deletion or compaction based on retention policy.
Replication factorEach partition is replicated across N brokers. One is the leader; the rest are followers. Minimum ISR size controls durability — if ISR shrinks below min.insync.replicas, produces with acks=all fail.
OffsetMonotonically increasing integer per partition. Consumers track their position by offset. Offsets are never reused — even deleted records leave gaps.
Retention policiesdelete — remove segments older than retention.ms or larger than retention.bytes. compact — keep only the latest record per key (good for changelogs).
Partition key routingRecords with the same key always land in the same partition, preserving order per key. Null keys are distributed round-robin across partitions.

Group Coordinator: What it is & what happens when it dies
What is a Group Coordinator?A regular broker elected to manage a specific consumer group. It handles JoinGroup, SyncGroup, Heartbeat, and OffsetCommit requests. Which broker is the coordinator is determined by: hash(group-id) mod __consumer_offsets partitions.
When crashed broker is NOT the coordinatorConsumer heartbeats keep flowing normally to the surviving coordinator broker. The only impact is partition leader re-election if the broker held partition leaders — consumers get a metadata refresh and reconnect automatically within milliseconds.
When crashed broker IS the coordinatorConsumers receive NOT_COORDINATOR on their next heartbeat or request. They call FindCoordinator which returns the new coordinator (another broker). A full group rebalance then kicks off — all partitions are temporarily unassigned and reassigned.
When crashed broker is ALSO the Kafka ControllerThe most disruptive case. ZooKeeper deletes the /controller znode; remaining brokers race to become controller. The new controller then runs all partition leader elections before normal operations resume. Typically takes 10–30 s.

💀 What happens when the ZooKeeper Leader fails?
%%{init: { "theme": "base", "themeVariables": { "primaryColor": "#3c3836","primaryTextColor": "#ebdbb2", "primaryBorderColor": "#665c54","lineColor": "#fb4934", "secondaryColor": "#504945","background": "#282828", "nodeBorder": "#cc241d","clusterBkg": "#1d2021", "clusterBorder": "#cc241d","titleColor": "#fb4934", "edgeLabelBackground": "#282828", "fontFamily": "JetBrains Mono, Fira Code, monospace" } }}%% flowchart TD classDef dead fill:#cc241d,stroke:#fb4934,color:#fbf1c7,font-weight:bold classDef zk fill:#b16286,stroke:#d3869b,color:#1d2021,font-weight:bold classDef zknew fill:#d79921,stroke:#fabd2f,color:#1d2021,font-weight:bold classDef step fill:#3c3836,stroke:#665c54,color:#ebdbb2 classDef broker fill:#504945,stroke:#bdae93,color:#ebdbb2 classDef ok fill:#98971a,stroke:#b8bb26,color:#1d2021,font-weight:bold DEAD["ZK Leader crashes"]:::dead DETECT["Followers detect silence after syncLimit x tickTime eg. 2 x 2s = 4s"]:::step ELECT["FastLeaderElection begins Exchange LOOKING messages Highest zxid epoch sid wins"]:::step QUORUM{"Quorum reached? majority > n/2"} NEW_LEADER["New ZK Leader elected highest zxid follower"]:::zknew SYNC["Followers sync to new leader DIFF / SNAP / TRUNC"]:::step NOTE1["In-flight uncommitted writes rolled back - truncated"]:::step REOPEN["New leader reopens all client sessions"]:::step KAFKA_CTRL["Kafka Controller detects ZK session change watches /controller znode"]:::broker RESUME["Cluster resumes normal operation"]:::ok STALE["No quorum ZK cluster unavailable"]:::dead DEAD --> DETECT --> ELECT --> QUORUM QUORUM -->|"majority alive"| NEW_LEADER QUORUM -->|"minority only"| STALE NEW_LEADER --> NOTE1 NEW_LEADER --> SYNC --> REOPEN --> KAFKA_CTRL --> RESUME
DetectionFollowers notice missing Zab pings after syncLimit × tickTime (default 4 s). They transition to LOOKING state.
FastLeaderElectionEach node votes for the candidate with the highest (zxid, epoch, sid) — highest zxid means most up-to-date transaction log.
Log sync (DIFF/SNAP/TRUNC)New leader syncs followers. Uncommitted writes (not quorum-acked before crash) are truncated — Zab guarantees no data loss for committed transactions.
Kafka impactKafka brokers briefly lose ZK connectivity. The Kafka Controller re-validates its /controller znode. If lost, a new Kafka Controller is elected. Typically resolves in < 30 s.

Consumer → Group Coordinator HeartbeatRequest sent on a background thread. Interval ≈ session.timeout.ms / 3. Missing the deadline triggers a group rebalance and partition reassignment.
Broker → ZooKeeper (classic mode) TCP session ping every tickTime (default 2 s). Expiry deletes the ephemeral /brokers/ids/<id> znode, signalling the controller to run leader election.
Broker → Active Controller (KRaft) BrokerHeartbeatRequest sent by every broker to the active KRaft controller, carrying the broker's epoch and log end offset. Replaces ZooKeeper session keep-alives in KRaft mode.
ZooKeeper Node → ZooKeeper Node (Zab) ZK nodes exchange PING/PONG messages every tickTime via the Zab protocol. The ZK leader also sends periodic heartbeats to followers. If a follower misses syncLimit ticks from the leader, it disconnects and FastLeaderElection kicks off a new ZK leader vote among the surviving quorum members.
Missed heartbeat consequences Consumer miss → group rebalance. Broker miss (ZK) → leader election. Broker miss (KRaft) → broker fenced. ZK node miss → ZK leader re-election via FastLeaderElection.
