PartitionsUnit of parallelism. Count can only increase, never decrease. More partitions = higher throughput but more file handles.
SegmentsEach partition splits into rolling .log + .index files. Older segments are deleted or compacted per retention policy.
Replication factorEach partition replicated across N brokers. If ISR shrinks below min.insync.replicas, acks=all produces fail.
OffsetMonotonically increasing integer per partition. Never reused — deleted records leave gaps.
Retentiondelete — remove by age/size. compact — keep latest record per key (changelogs).
Key routingSame key → same partition → ordering per key. Null key → round-robin across partitions.

FastLeaderElection — triggered when followers miss heartbeats
%%{init:{"theme":"base","themeVariables":{"primaryColor":"#3c3836","primaryTextColor":"#ebdbb2","primaryBorderColor":"#665c54","lineColor":"#fb4934","secondaryColor":"#504945","background":"#282828","nodeBorder":"#cc241d","clusterBkg":"#1d2021","clusterBorder":"#cc241d","titleColor":"#fb4934","edgeLabelBackground":"#282828","fontFamily":"JetBrains Mono, Fira Code, monospace"}}}%% flowchart TD classDef dead fill:#cc241d,stroke:#fb4934,color:#fbf1c7,font-weight:bold classDef step fill:#3c3836,stroke:#665c54,color:#ebdbb2 classDef dec fill:#504945,stroke:#fabd2f,color:#fabd2f,font-weight:bold classDef ok fill:#98971a,stroke:#b8bb26,color:#1d2021,font-weight:bold classDef zknew fill:#d79921,stroke:#fabd2f,color:#1d2021,font-weight:bold DEAD["ZK Leader stops sending heartbeats"]:::dead DETECT["Followers detect silence / after syncLimit x tickTime / default 2 x 2s = 4s"]:::step LOOK["Followers transition to LOOKING state"]:::step VOTE["Each node broadcasts vote for itself / tuple: (zxid, epoch, sid)"]:::step COMPARE["On receiving peer vote / compare: epoch then zxid then sid / Higher wins — node re-votes with winner"]:::step ADOPT["If vote has newer epoch / node adopts epoch and re-votes"]:::step QUORUM{"sid voted for by / majority of quorum?"}:::dec NEW_LEADER["New ZK Leader elected / highest (epoch, zxid, sid)"]:::zknew SYNC["Followers sync to new leader / DIFF / SNAP / TRUNC / uncommitted writes rolled back"]:::step KAFKA["Kafka Controller detects / /controller znode change / Re-elects Kafka Controller if needed"]:::step RESUME["Normal operations resume"]:::ok STALE["No quorum — ZK unavailable / Kafka brokers lose metadata access"]:::dead DEAD --> DETECT --> LOOK --> VOTE --> COMPARE --> ADOPT --> QUORUM QUORUM -->|YES — majority alive| NEW_LEADER QUORUM -->|NO — minority only|STALE NEW_LEADER --> SYNC --> KAFKA --> RESUME
TriggerFollowers miss syncLimit × tickTime (default 4 s) of Zab pings and transition to LOOKING state.
Vote comparison orderepoch < zxid < sid — highest wins. If a node receives a vote with a newer epoch, it adopts that epoch and re-votes.
Log sync after electionDIFF for small gaps, SNAP for full resync, TRUNC to roll back uncommitted writes. Zab guarantees committed transactions are never lost.
Kafka impactKafka Controller re-validates /controller znode. If lost, a new Kafka Controller is elected. Typically resolves in < 30 s.

Offset keyOffsets are stored as the triple (group.id, topic, partition) in both the in-memory table and __consumer_offsets.
In-memory mapAt startup the Group Coordinator builds a map from __consumer_offsets. Subsequent reads/writes use the in-memory map — not the topic.

Partition sizingIf a topic has X partitions, use ≥ X consumers in the group. Extra consumers sit idle; fewer consumers means one consumer handles multiple partitions.

Broker Failure — Group Coordinator

What is the Group Coordinator?
DefinitionA broker elected to manage a consumer group. Handles JoinGroup, SyncGroup, Heartbeat, OffsetCommit. The coordinator for a group is determined by hash(group.id) mod __consumer_offsets_partitions.
Co-locationEach Group Coordinator is also the partition leader of the __consumer_offsets partition that stores that group's offsets. They are the same broker.

Side-by-side: coordinator dead vs not coordinator
Broker is NOT coordinatorConsumer heartbeats keep flowing to the surviving coordinator. Only impact is partition leader re-election. Consumers auto-refresh metadata and reconnect within milliseconds.
Broker IS coordinatorConsumers get NOT_COORDINATOR. Call FindCoordinator → new broker. Full group rebalance. All partitions unassigned then re-assigned. Consumption pauses briefly.
Broker is also Kafka ControllerMost disruptive. ZooKeeper deletes /controller znode, remaining brokers race to become controller, then all partition leader elections run before normal operations resume. Typically 10–30 s.

Consumer → Group CoordinatorBackground thread. Interval ≈ session.timeout.ms / 3. Miss triggers group rebalance and partition reassignment.
Broker → ZooKeeperTCP session ping every tickTime (2 s). Expiry deletes ephemeral /brokers/ids/<id> znode → controller runs leader election.
Broker → Active Controller (KRaft)BrokerHeartbeatRequest carrying epoch and log end offset. Replaces ZK session keep-alives in KRaft mode. Miss fences the broker.
ZK Node ↔ ZK Node (Zab)PING/PONG every tickTime. Follower misses syncLimit ticks → disconnects → FastLeaderElection among surviving quorum.
Consequences at a glanceConsumer miss → group rebalance. Broker miss (ZK) → leader election. Broker miss (KRaft) → broker fenced. ZK node miss → ZK leader re-election.

Component Legend

ProducerPublishes records to topic-partition leaders. Handles batching, compression, and retries.
Partition LeaderReceives all writes for a partition. Manages high-watermark and ISR. Serves consumer fetches.
Follower / ReplicaPassively replicates from leader. Promoted to leader on failure if in ISR.
ConsumerPulls records by offset. Commits offsets to track progress. Belongs to a consumer group.
Consumer GroupSet of consumers sharing a group-id. Each partition assigned to at most one consumer per group.
ZooKeeper / KRaftManages cluster metadata, broker liveness, controller election, config storage. Uses ZAB protocol for writes.
ControllerElected broker that orchestrates leader elections, ISR changes, and partition reassignments.
Group CoordinatorBroker that manages a consumer group. Also the partition leader of the group's __consumer_offsets partition.
__consumer_offsetsInternal compacted topic. Stores committed offsets keyed by (group.id, topic, partition).
